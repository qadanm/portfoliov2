## The platform

MagTek makes payment hardware and software — card readers, PIN pads, secure payment terminals. Their primary website is the primary surface for a broad audience: enterprise buyers evaluating hardware, developers integrating APIs, partners navigating support, and end users troubleshooting devices.

I own and evolve the UX for approximately 85% of this production website, plus maintain UX support across 11 additional properties that share the same architecture.

## What "platform ownership" actually means

This is not a feature project. It's not a redesign. It's sustained, production-scale UX work inside a real, long-lived codebase.

That means:

- Understanding how pages relate to each other at a system level, not just page-by-page
- Making UX decisions that account for shared layouts, navigation state, and component behavior across sections
- Working within real constraints — an ASP.NET MVC 5 monolith, Razor views, jQuery and Bootstrap, Entity Framework, IIS — rather than starting from a clean greenfield
- Improving things in place, without breaking what's already working
- Writing changes that a future maintainer can understand without a written guide

This kind of work is slower than building something new, harder to explain in a portfolio, and significantly more valuable than it looks.

## Technical environment

The stack is a real production stack, not a modern framework stack:

- **ASP.NET MVC 5 / .NET Framework** — Razor views, controller-based routing, model binding
- **Bootstrap + jQuery** — front-end layer; improvements require working within or carefully around existing patterns
- **Entity Framework** — data access layer that intersects with frontend routing decisions
- **Web API 2** — internal APIs that feed dynamic content and integrations
- **IIS** — hosting with real staging/production distinction and deploy process
- **SMTP workflows** — transactional flows that touch UX at critical touchpoints

Working in this environment teaches you things a greenfield project cannot. You learn how decisions made years ago echo into current constraints, how shared templates create coupling that limits change, and how the right improvement is often not the obviously correct one but the one that doesn't break four other things.

## What I actually do

**Component and layout consistency** — Identifying where shared layout patterns diverge across sections and bringing them toward coherence. This is not glamorous work. It is what keeps a large site from fragmenting into a collection of disconnected pages.

**Page-level UX improvements** — Product pages, support pages, service landing pages, onboarding flows. Each one improved within the constraints of the shared architecture rather than rebuilt from scratch.

**Navigation and routing structure** — Understanding how users move through the site and where the information architecture creates friction. Making routing changes that improve discoverability without breaking existing links or SEO structure.

**System reliability and consistency** — When a template change propagates across 40 pages, getting it right the first time matters. I work with that reality, not against it.

**Shared component evolution** — Taking components that were built for one section and improving them in ways that work when they appear in other contexts. This requires understanding the whole before changing the part.

## What this isn't

This isn't a startup that rebuilds its frontend every 18 months. The codebase predates many modern frameworks. The constraints are real.

That's the point. Most of the interesting, high-value work in software happens inside systems like this — not in greenfield projects with perfect stacks. Understanding how to improve something complex without breaking it is a different skill than building something new from scratch.
